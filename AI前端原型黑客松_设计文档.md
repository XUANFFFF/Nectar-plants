# AI 前端原型黑客松设计文档
## AI Coding Prototype Hackathon

> 文档性质：黑客松活动设计 / 参赛说明书  
> 版本：v1.0  
> 适用对象：项目小伙伴、业务同事、AI 编程初学者  
> 核心定位：用虚拟业务数据包 + AI 编程工具，在 2–3 天内做出一个可展示的前端交互平台原型。

---

## 一、活动定位

这次黑客松不是传统意义上的“算法比赛”，也不是要求大家从零写出一个完整系统。它更像一次 AI 辅助产品原型工作坊：每个人基于同一份虚拟 Excel 数据包，使用 Codex、Trae、Cursor、Claude Code 或其他 AI 编程工具，快速做出一个前端平台 Demo。

活动重点不在于代码多复杂，而在于三件事：

第一，能不能理解数据字段背后的业务场景。  
第二，能不能把业务需求转化为清晰的页面结构和交互功能。  
第三，能不能用 README 把自己的设计思路、开发过程和结果讲明白。

最终大家会在分享会上展示自己的平台原型、设计思路和 README 文件。不同作品可以风格完全不同：有人做成数据看板，有人做成 AI 助手，有人做成需求管理平台，有人做成素材库或分析中台，都可以。

---

## 二、活动目标

本次黑客松有四个目标：

1. 让团队成员真实体验 AI 编程工具如何参与产品开发，而不是停留在“让 AI 写一段代码”的浅层使用。
2. 通过统一的虚拟数据包，让大家围绕同一批字段做出不同理解，激发对业务功能的创造力。
3. 沉淀一批可参考的前端平台原型，为后续真实项目提供页面结构、交互方式和功能设计参考。
4. 训练每个人用 README 记录自己的开发逻辑，形成可复盘、可传播、可复用的过程资产。

一句话总结：这次不是比“谁最会写代码”，而是比“谁最会把业务问题变成一个可以被看见、被点击、被理解的产品原型”。

---

## 三、参与对象与前置要求

参与者不需要是专业程序员，但需要具备基本的电脑操作能力，并愿意使用 AI 编程工具进行尝试。

推荐参与者提前准备：

- 一台个人电脑；
- 一个 AI 编程工具账号，如 Codex、Trae、Cursor、Claude Code 等；
- 一个代码运行环境，可以是本地开发环境，也可以是在线平台；
- 基础的 GitHub 或文件夹管理习惯；
- 愿意用 README 记录过程，而不是只交一个最终页面截图。

如果参与者不会写代码，也可以参加。建议选择 HTML 单页、Streamlit、React 模板改造或低代码工具路线。黑客松的底层原则是：先跑通，再优化；先可展示，再谈高级。

## 五、开发任务说明

### 5.1 核心任务

请基于虚拟 Excel 数据包，使用 AI 编程工具开发一个可交互的前端平台原型。

平台可以是（不限于）以下方向之一，也可以自由组合：

- 数据总览看板；
- 项目需求管理平台；
- 内容素材库；
- AI 助手式数据查询页面；
- 报告生成工具；
- 其他你认为更有价值的业务平台。

### 5.2 最低功能要求

最终 Demo 至少需要满足以下要求：

1. 使用 Excel 数据包中的工作表。
2. 页面中至少包含 3 类交互功能，例如搜索、筛选、排序、详情弹窗、卡片展示、图表、状态标记、导出按钮、AI 总结区域等。
3. 页面要能体现你对“项目需求方可能需要什么功能”的理解，而不是只把表格原样贴出来。
4. 提交时必须包含 README 文件，说明你做了什么、为什么这样做、怎么运行、还有什么待优化。

### 5.3 不做强限制的部分

为了鼓励自主创作，以下内容不做强制要求：

- 不限制技术栈，可以用 React、Vue、Next.js、HTML/CSS/JS、Streamlit、Gradio、低代码平台等。
- 不限制页面风格，可以是咨询风、SaaS 风、BI 看板风、Notion 风、Codex 风或其他风格。
- 不限制功能方向，只要能基于数据包和业务理解展开即可。
- 不要求接入真实后端，前端静态模拟也可以。
- 不要求部署上线，本地可运行或录屏演示也可以。

## 六、推荐时间安排

本次活动建议采用 2–3 天个人开发 + 1 次分享会议的节奏。

### Day 0：准备与资料发放

组织方发放设计文档、技术资源包和虚拟 Excel 数据包。参与者完成工具安装、阅读任务说明、初步选择方向。

建议参与者在 Day 0 完成三件事：

1. 打开 Excel 数据包，理解每个 Sheet 的字段。
2. 选择一个自己想做的产品方向。
3. 用 AI 工具生成第一版页面结构草图或功能清单。

### Day 1：选题与原型搭建

目标是完成最小可运行版本。

推荐节奏：

- 30 分钟：确定目标用户和使用场景；
- 30 分钟：让 AI 帮你生成产品 PRD 或页面结构；
- 1–2 小时：搭建页面骨架；
- 1–2 小时：接入或模拟 Excel 数据；
- 30 分钟：记录 README 的“项目目标”和“功能设计”。

Day 1 不要追求完美。只要页面能打开、数据能展示、交互能点击，就已经成功一半。

### Day 2：功能完善与 README 整理

目标是把 Demo 从“能跑”改成“能讲”。

推荐节奏：

- 完善筛选、搜索、卡片、详情、图表等功能；
- 统一视觉风格；
- 写清楚 README；
- 保留关键过程文件，例如 Prompt 记录、截图、开发日志；
- 准备 3–5 分钟演示讲稿。

### Optional Day 3：优化与彩排

如果活动安排为 3 天，可以把第三天用于优化：

- 修复明显 Bug；
- 优化页面布局；
- 补充 README 截图；
- 准备演示视频；
- 形成“我学到了什么”的复盘。

每位参与者或每个小组最终提交一个文件夹，建议结构如下：

```text

├── README.md

│   ├── 页面截图.png（可选）
│   └── 演示录屏.mp4（可选）

│   └── 使用的数据样例或字段说明
└── process/
    ├── prompt_log.md
    ├── design_notes.md
    └── issue_log.md
```

如果不方便提交完整代码，也至少提交：

1. README.md；
2. 关键 Prompt 记录；
4. 项目文件或在线链接。

## 九、README 必须包含的内容（可自由发挥，框架仅供参考）

README 是本次交付的核心部分之一。建议至少包含：

1. 项目名称；
2. 一句话介绍；
3. 目标用户；
4. 解决的业务问题；
5. 使用的数据表；
6. 核心功能；
7. 页面结构；
8. 技术栈；
9. 运行方式；
10. AI 工具使用过程；
11. 主要 Prompt；
12. 遇到的问题与解决方式；
13. 下一步优化方向。

README 不需要写得像论文，但要让别人看完后知道：你为什么做这个、怎么做出来的、它能解决什么问题。

## 十、分享会议安排

分享会议建议每人或每组 5–8 分钟：

- 1 分钟：介绍项目目标和目标用户；
- 2 分钟：演示页面和核心功能；
- 1 分钟：说明数据如何被使用；
- 1 分钟：说明 AI 编程工具如何帮助你；
- 1–3 分钟：评委或同伴提问。

如果参与者较多，可以分成两个环节：

第一轮快速 Demo，每人 3 分钟。  
第二轮重点作品深挖，每个优秀作品 8–10 分钟。

---

## 十一、评审标准

建议采用 100 分制。

| 维度 | 分值 | 评估问题 |
|---|---:|---|
| 业务理解 | 25 | 是否清楚理解目标用户、业务场景和真实需求？ |
| 数据使用 | 20 | 是否真正使用了数据包，而不是只做静态页面？ |
| 交互设计 | 20 | 页面是否清晰、好用、有基本产品体验？ |
| AI 编程过程 | 15 | 是否有效利用 AI 工具完成设计、开发和调试？ |
| README 质量 | 15 | 是否能清楚记录设计思路、运行方式和过程复盘？ |
| 创意加分 | 5 | 是否有独特视角或让人眼前一亮的功能？ |

本次评审不建议过度强调技术复杂度。对于业务同事和初学者来说，能把一个功能讲清楚、做出来、跑起来，本身就是很好的结果。

## 十三、给参与者的提醒

不要一开始就追求“大而全”。一个好的黑客松作品，通常不是功能最多，而是问题定义最清楚。

你可以先问自己四个问题：

1. 我这个平台给谁用？
2. 他打开页面后第一眼要看到什么？
3. 他最需要完成的一个动作是什么？
4. 我手里的 Excel 数据能如何支持这个动作？

把这四个问题想清楚，再让 AI 帮你写代码，效果会比直接说“帮我做一个网站”好很多。
