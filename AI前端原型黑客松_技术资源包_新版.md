## 一、推荐使用的 AI 工具

本次黑客松鼓励大家使用 AI 编程工具完成产品原型开发。工具不做强制限制，大家可以根据自己的电脑环境、账号权限、使用习惯和技术基础自由选择。

这里的工具推荐只用于降低启动门槛，不代表只能使用这些工具。你可以使用 Codex、Trae、Cursor、Claude Code，也可以使用 ChatGPT、Claude、Gemini、DeepSeek、Kimi、通义千问等大模型辅助完成需求分析、页面设计、代码生成、调试和 README 整理。

### 1.1 国外 AI 编程工具

| 工具 | 适合人群 | 可以帮你做什么 | 使用建议 |
|---|---|---|---|
| Codex | 希望用 AI 直接读代码、改代码、运行项目的参与者 | 生成页面、修改组件、解释报错、优化交互、整理项目结构 | 适合让它“分步骤完成任务”，例如先搭页面骨架，再接数据，再优化 README |
| Claude Code | 有一定命令行基础，愿意让 AI 在本地项目中协作开发的参与者 | 理解代码库、编辑文件、运行命令、调试问题、生成文档 | 适合已有项目文件夹后使用，让它持续迭代，而不是只写一次代码 |
| Cursor | 熟悉 VS Code，想在编辑器里边写边改的参与者 | 代码补全、组件生成、局部修改、错误解释、项目重构 | 适合前端初学者边看代码边让 AI 修改 |
| GitHub Copilot | 已经使用 GitHub 或 VS Code 的参与者 | 代码补全、函数生成、局部问答 | 适合写小功能和补充细节，不一定适合独立完成完整项目 |

### 1.2 国内 AI 编程工具

| 工具 | 适合人群 | 可以帮你做什么 | 使用建议 |
|---|---|---|---|
| Trae | 想快速从想法生成软件原型的参与者 | 根据任务描述生成应用、修改代码、解释项目结构、辅助调试 | 适合用自然语言描述“我要做一个什么平台”，再逐步让它拆页面和功能 |
| 通义灵码 | 使用国内开发环境或阿里云生态的参与者 | 代码补全、代码解释、单元功能生成 | 适合在编辑器中辅助开发，不一定承担完整产品设计 |
| 豆包 / DeepSeek / Kimi / 通义千问 | 不想一开始进入复杂 IDE 的参与者 | 需求拆解、字段理解、页面方案、Prompt 生成、README 润色 | 适合先让模型帮你想清楚“做什么”，再把方案交给编程工具实现 |
| Coze / Dify / 扣子空间等低代码工具 | 更偏产品、运营、业务背景的参与者 | 搭建 AI 助手、流程 Demo、对话式页面或知识库应用 | 如果不擅长写前端，可以用这些工具做“交互逻辑原型” |

### 1.3 其他可用模型与工具

如果你不想使用完整的 AI 编程工具，也可以把大模型当作“产品经理 + 前端老师 + 调试助手”来使用。推荐的使用方式是：

第一步，先让 AI 读懂数据表字段，并帮你提出 3–5 个产品方向。  
第二步，选择一个方向，让 AI 帮你写页面结构和功能清单。  
第三步，再让 AI 生成 HTML / React / Vue / Streamlit 等代码。  
第四步，把报错截图或终端报错贴回 AI，让它协助你修复。  
第五步，让 AI 帮你整理 README、演示讲稿和复盘总结。

你可以自由使用以下类型工具：

| 类型 | 可选工具 | 适合用途 |
|---|---|---|
| 通用大模型 | ChatGPT、Claude、Gemini、DeepSeek、Kimi、通义千问 | 需求分析、页面规划、代码解释、README 写作 |
| AI 编程 IDE | Codex、Trae、Cursor、Claude Code、GitHub Copilot | 代码生成、项目搭建、调试、迭代 |
| 前端框架 | HTML/CSS/JS、React、Vue、Next.js | 做可点击的网页原型 |
| 快速 Demo 工具 | Streamlit、Gradio | 用 Python 快速做数据展示和交互页面 |
| 低代码 / 工作流工具 | Dify、Coze、飞书多维表格、Notion、Airtable | 做 AI 助手、流程演示、数据管理原型 |
| 文档工具 | README、飞书文档、Notion、Markdown | 记录思路、过程、结果和复盘 |

### 1.4 推荐使用原则

本次黑客松不考察“谁用的工具最高级”，而是看你是否能用工具把想法做出来。建议大家遵循以下原则：

不要一开始就追求完整系统。先做一个能打开、能点击、能展示数据的版本。  
不要把 AI 当成一次性代码生成器。更好的方式是持续对话、不断纠错、逐步迭代。  
不要只说“帮我做一个网站”。要告诉 AI：目标用户是谁、数据字段有哪些、页面要解决什么问题、希望有哪些交互。  
不要害怕作品简单。一个清楚、稳定、能讲明白的小 Demo，比一个跑不起来的大系统更有价值。

### 2.7 如何使用这些参考页面

建议大家不要只看“页面好不好看”，而是重点观察这几个问题：

这个网站的首页第一屏想让用户理解什么？  
它如何组织复杂信息：用卡片、列表、地图、详情页，还是专题叙事？  
它有哪些筛选、搜索、跳转、地图、标签、图表等交互？  
如果我要借鉴一个模块，哪个模块最适合我的数据包？  
我能不能把它简化成 2–3 天能做出来的小版本？

一个简单的借鉴方式是：

```text
我参考了 iNaturalist 的观察记录思路，但我的作品不做完整社区，只做一个“观察数据卡片 + 地图点位 + 物种筛选”的轻量原型。
```

或者：

```text
我参考了植物多样性物种详情页的信息结构，把虚拟昆虫数据做成“列表页 + 详情弹窗 + 标签筛选”的前端 Demo。
```

## 结语

这个技术资源包只提供起点，不提供标准答案。你可以参考工具、参考网站、参考数据字段，但最终作品应该体现你自己的业务理解和产品判断。

本次黑客松最重要的不是“做得多复杂”，而是：你能不能把一个模糊的业务想法，借助 AI 编程工具，变成一个能打开、能点击、能讲清楚的前端原型。
