# 深圳城市绿地传粉动物公民科学项目

## 一句话介绍

这是一个面向城市自然观察者、志愿者和项目参与者的公民科学项目网页平台。它把蜜源植物观察数据转化为可交互的城市地图、植物图鉴、伙伴动态、专家解读和资料库，让参与者看见自己上传的数据如何进入公共知识库，并了解项目成果和科学发现。

## 在线预览

线上预览地址：https://nectar-plants.pages.dev/

## 项目定位

本平台从"城市蜜源共创地图原型"升级为"深圳城市绿地传粉动物公民科学项目平台"，核心变化：

- **持续展示**：代替过去每年一份静态报告的形式，全年持续更新数据展示
- **参与反馈**：参与者实时看到自己的贡献如何形成公共知识
- **专家层叠**：新增专家解读和项目成果模块，区分实时数据与正式科学结论
- **资料沉淀**：历年报告、使用指南、识别手册统一归集

平台与年度报告的关系：网页负责持续展示与参与反馈，年度报告/专家模块负责正式解读和沉淀。

## 数据实时性说明

**当前状态**：本项目支持两种数据模式：

1. **静态 demo 模式**（默认）：使用 `src/data/observations.json` 等本地 JSON。
2. **数据库模式**：启动后端服务后，前台自动从 API 读取数据；API 不可用时回退到本地 JSON。

启动数据库模式：`npm run dev:all`（同时启动后端 API 服务器和前端开发服务器）。

### 数据模式切换

前台 `src/lib/data-service.ts` 会自动检测 API 是否可用：
- API 可用 → 从数据库读取真实数据
- API 不可用 → 回退到本地静态 JSON

页面数据状态栏和页脚会显示当前数据源。

## 目标用户

- 城市自然观察活动的志愿者：希望知道自己的记录被如何使用、产生了什么价值。
- 项目参与者与运营同事：希望快速了解不同城市、区县、植物和访花昆虫的共创情况。
- 活动展示或科普传播对象：希望用更轻松的方式理解蜜源植物数据，而不是直接阅读表格。
- 科研与合作机构：希望了解项目进展、数据概览和阶段性发现。

## 使用的数据

数据源为 `志愿者上传数据_demo虚拟数据_170条_15名志愿者_非均分.xlsx`，经脚本整理后生成前端可直接读取的 JSON。

当前原型使用的数据规模：

- 200 条观察记录
- 19 位共创伙伴
- 25 种蜜源植物
- 19 个科
- 4 个城市
- 32 个区县
- 113 个观察场域

主要使用字段包括：

- 物种信息：中文名、学名、科、属、来源类型
- 空间信息：省、市、区县、经纬度、海拔
- 观察信息：观察者、观察日期、识别日期、备注
- 共创信息：访花昆虫、图片链接、审核状态、数据集名称、许可证

更完整的字段说明见 `data/field_notes.md`。

## 页面结构

- `#home`：项目首页 — 数据规模、最新观察、数据状态与质量提示
- `#map`：蜜源地图 — 城市边界与观察点位分布
- `#plants`：植物图鉴 — 按植物聚合记录
- `#activity`：伙伴动态 — 最新观察、城市亮点、植物-访花故事
- `#me`：我的贡献 — 个人贡献统计与徽章
- `#insights`：专家解读 — 阶段性发现、项目成果与社会影响
- `#resources`：资料库 — 历年报告、使用指南、识别手册

## 核心功能

1. 项目首页

   展示整体数据规模、最新观察记录、数据更新时间、来源和审核状态。首页明确区分实时数据（持续更新）与专家解读（阶段性发布）。

2. 蜜源地图

   基于广东省城市边界和观察点位展示数据分布。用户可以点击城市边界查看该城市的记录数、场域数、物种数、重点区县和代表植物。地图支持按全部点位、野生蜜源、人工养育、近 30 天记录切换图层。

3. 植物图鉴

   按植物聚合观察记录，展示真实图片、中文名、学名、科属、记录数。点击植物卡片后，可以查看该植物出现的城市、区县、访花关联和上传证据。

4. 伙伴动态

   展示最新观察、城市亮点和植物-访花昆虫故事，让数据具备"正在发生"的动态感。

5. 我的贡献

   顶部可以切换不同志愿者视角，查看该志愿者贡献的记录数、物种数、区县数、场域数和徽章。

6. 专家解读

   展示阶段性分析发现（数据概览、关键发现、访花网络分析），以及项目成果与社会影响（志愿者参与、花园改造、科学传播、合作网络）。页面明确标注"以下内容基于当前数据初步解读，不代表正式科学结论"。

7. 资料库

   历年报告、使用指南、识别手册等资料的归集展示，第一版为静态示例内容。

## 主要交互

- 导航切换：在项目首页、蜜源地图、植物图鉴、伙伴动态、我的贡献、专家解读、资料库之间切换。
- 志愿者视角切换：选择不同观察者，查看对应个人贡献。
- 城市地图筛选：点击城市边界筛选该城市的观察场域和汇总数据。
- 图层筛选：按全部、野生、人工养育、近 30 天记录查看地图。
- 植物详情弹窗：点击植物卡片查看分布、生境、访花关联和证据。
- 场域信息提示：到达或选中场域时，可查看该场域相关信息。
- 图片展示：使用 `associatedMedia` 中的真实图片链接，并缓存到本地 `public/media/`。

## 数据架构

当前数据链路：

```
Excel → Python 脚本 → src/data/observations.json / summary.json → data-service.ts → 前端组件
```

数据读取已抽象为 `src/lib/data-service.ts`，提供 `getObservations()`、`getSummary()`、`getMetadata()`、`getVerificationStats()`、`getDataQualityStats()` 等接口。前台优先尝试从 API 读取，失败时回退到本地 JSON：

```
API / 数据库 / BioGrid → data-service.ts → 前端组件（无需修改组件代码）
                                    ↕ (fallback)
                              src/data/*.json
```

## 后端服务与数据导入

### 启动方式

```bash
# 同时启动后端（端口 3001）和前端（端口 5173）
npm run dev:all

# 或分别启动：
npm run dev:server    # 后端 API
npm run dev           # 前端 Vite
```

前端 Vite 开发服务器自动代理 `/api/*` 到后端。

### 数据库

使用 **SQLite**（文件位于 `data/nectar.db`），零配置，自动创建。

表结构（`server/schema.sql`）：

- `observations`：观察记录表，包含所有标准化字段
- `import_batches`：导入批次记录
- `import_errors`：行级导入错误

### Excel 导入流程

管理后台入口：导航 → "管理"（`#admin`）

流程：

1. 选择 `.xlsx` / `.xls` 文件
2. 系统解析并返回预览（总行数、有效行、错误行、前 20 条记录、行级错误）
3. 用户确认导入
4. 系统去重后批量写入数据库
5. 前台自动从数据库读取最新数据

### 字段映射

Excel 表头到标准字段的映射配置见 `server/field-mapping.js`。支持中文和英文表头，当前要求至少包含以下必填字段：

- 观察日期 / `event_date`
- 中文名 / `chinese_name` 或学名 / `scientific_name`
- 城市 / `city`
- 观察者 / `observer`

### 去重策略

使用 `source_record_id` 或 `observer + event_date + chinese_name/scientific_name + longitude + latitude + media_url` 生成稳定 hash 进行去重。

### 管理后台权限

当前使用环境变量 `ADMIN_KEY` 进行简单鉴权（默认 `dev-key-change-me`），写入接口需要 `X-Admin-Key` 请求头。生产环境部署前必须增加完整的用户认证系统。

### API 接口列表

| 方法 | 路径 | 说明 | 鉴权 |
|------|------|------|------|
| GET | `/api/observations` | 获取观察记录 | 无 |
| GET | `/api/observations/:id` | 获取单条记录 | 无 |
| GET | `/api/summary` | 获取聚合统计 | 无 |
| GET | `/api/metadata` | 获取元数据（更新时间、数据源） | 无 |
| GET | `/api/verification-stats` | 获取审核统计 | 无 |
| GET | `/api/data-quality` | 获取数据质量统计 | 无 |
| POST | `/api/imports/upload` | 上传并解析 Excel | Admin Key |
| POST | `/api/imports/:id/confirm` | 确认导入 | Admin Key |
| GET | `/api/imports` | 导入批次列表 | 无（仅查询） |
| GET | `/api/imports/:id` | 批次详情 | 无（仅查询） |
| POST | `/api/imports/:id/cancel` | 取消批次 | Admin Key |
| POST | `/api/sync/observations` | 外部系统同步（预留） | 需鉴权（未实现） |

## 数据处理方式

Excel 数据通过 `scripts/extract-observations.py` 转换为：

- `src/data/observations.json`：每条观察记录的结构化数据
- `src/data/summary.json`：整体统计摘要
- `public/media/`：从 `associatedMedia` 链接缓存下来的图片

前端再基于这些 JSON 数据进行聚合：

- 按城市、区县、观察场域聚合成地图数据
- 按植物名称聚合成图鉴数据
- 按观察者聚合成个人贡献数据
- 按访花昆虫字段生成植物-昆虫关联故事

## 遇到的问题与解决方式

1. 地图点位和城市位置对不上

   解决方式：统一城市边界和观察点的投影逻辑，保证经纬度点位落在正确地图范围内。

2. 植物图鉴没有使用真实图片

   解决方式：解析 `associatedMedia` 字段，下载并缓存可用图片，植物卡片优先展示真实观察图片。

3. 图片尺寸不一导致卡片高度不统一

   解决方式：固定图鉴卡片图片区域高度，使用 `object-fit: cover` 保持视觉一致。

4. 观察点太多导致无法点击城市

   解决方式：把点位调整为视觉热度层，关闭点位的鼠标拦截，让城市边界成为主要点击目标。

5. 原型容易偏向后台审核

   解决方式：把产品叙事改为"共创地图 + 个人贡献 + 动态故事"，强化参与者视角。
